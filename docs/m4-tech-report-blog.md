# 세무 AI는 "잘 말하는가"가 아니라 "검증을 통과했는가"

> K-TaxBench 기술 리포트 블로그 요약본 (발행처: K-TaxBench 자체 채널 또는 askewly 블로그 — 미정).

> 상태: 초고(v0.1, 2026-06-12).

회계법인이 세무 AI를 도입할지 말지 결정해야 한다고 하자. 벤더가 "우리 모델은 세무사 시험 정답률 90%"라고 한다. 이 숫자로 무엇을 알 수 있을까?

거의 없다. 실무에서 치명적인 오류는 "틀린 답"이 아니라 **그럴듯하지만 검증되지 않은 답**이기 때문이다. 정답률 한 숫자는 다음 다섯 가지를 구분하지 못한다.

- 답은 맞지만 **근거가 틀린** 경우
- 계산은 맞지만 **실무 설명이 위험한** 경우
- 조문은 찾았지만 **사실관계 적용이 틀린** 경우
- **개정세법을 반영하지 못한** 경우
- 불확실한 사안을 **단정한** 경우

세무사·회계사가 검토 단계에서 매일 잡아내는 게 바로 이런 것들이다. 객관식 정답률은 이걸 못 본다.

**K-TaxBench**는 이 다섯 실패 모드를 분해해 측정하는 평가 인프라다. 핵심 명제는 한 줄이다 — 세무 AI가 신뢰를 얻으려면 "잘 말한다"가 아니라 **"검증을 통과했다"**가 필요하다.

## 회계가 이미 답을 알고 있었다

흥미로운 출발점은, "신뢰할 수 있는 정보"의 조건을 회계가 이미 정의해 뒀다는 것이다 — 「재무보고를 위한 개념체계」의 질적 특성이다. **재무정보에 요구하는 질적 특성을 AI 평가 정보에도 똑같이 요구한다**는 것이 이 벤치마크의 설계 원리다.

- 목적적합성 → 점수 1점 차이가 "실무에 1만큼 더 믿고 맡길 수 있음"을 의미해야 한다(구성 타당도)
- 표현충실성 → 정답이 법령·기준서의 실질을 충실히 표현해야 한다(전문가 검수 + 기준일 도장)
- 검증가능성 → 누가 채점해도 같은 점수, 학습 누출이 아닌 실제 능력(오염 저항 + 자동채점)
- 적시성 → 세법은 매년 바뀐다(시점 버전 관리)
- 이해가능성 → "총점 78점"이 아니라 *어디가 약한지*를 보여줘야 한다(다차원 진단)

"회계 전문가가 설계한 AI 평가"라는 정체성이 여기서 나온다.

## 무엇을 어떻게 재는가

101문항을 6개 도메인(부가세·법인세·소득세·회계처리·기초세법·복합) × 7개 과제유형으로 분해하고, 답변을 **정확성·근거성·실무성·리스크 인식·도구 사용성** 5축으로 채점한다. 같은 문항을 closed-book / RAG / agent 세 모드로 돌려 *도구 사용이 신뢰도에 주는 효과*를 분리한다.

채점은 두 층이다. 객관식·계산·근거 위치는 **코드로 결정론 채점**하고, 사례·근거의 질은 LLM-judge가 7차원 부분점수로 채점한다. 단, 평가받는 모델이 자기 답을 채점하면 천장이 생기므로 **judge는 항상 후보군 밖 모델**로 두고, 후보·judge 실행 환경도 격리한다.

## 라이브로 확인한 세 가지

**① 모델 사이에 변별이 살아있다 — spread 40.2점.** opus 92.0 · sonnet 86.3 · haiku 51.7(공통 118쌍). 직전 30.1에서 확대됐다. 신규 소득세 hard 문항이 약모델을 더 강하게 갈랐다. 다 만점이 되어 변별이 사라지는(saturation) 방향의 반대 — 헤드룸이 남아있다는 신호다.

**② RAG는 정답을 알려주기보다 환각을 막아 점수를 올린다 — +8.6점.** 법령 원문을 주입하니 평균 72.4 → 81.0. 그 메커니즘을 들여다보면, 약모델(haiku)이 *존재하지 않는 근거를 지어내는* fake_source가 14건 → 4건으로 71% 줄었다. 다만 RAG가 만능은 아니다. 답이 곧 조문인 문항에선 크게 오르지만, 정당한 사유·절차·심판례 판단이 필요한 국세기본법 문항에선 오히려 떨어졌다 — 조문만 주입하면 모델이 엉뚱하게 앵커링되기 때문이다. *모드별 차이를 분해해 보여주는 것 자체*가 이 벤치마크가 정답률 하나를 넘는 지점이다.

**③ 도구 사용은 "강제 + 근거 매칭"으로만 신뢰성 있게 측정된다.** "답이 외부 사실을 필요로 하면 모델이 도구를 쓸 것"이라는 가설은 틀렸다. 유명한 한도(감가상각·운행기록)를 묻자 세 모델 모두 도구를 한 번도 안 썼다 — 암기하고 있으니까. 더 날카로운 장면: 강제 게이트를 걸었더니 약모델은 도구를 0회 호출하면서도 **조문 원문을 토씨까지 정확히 인용하며 "도구 확인 완료"라 적었다 — 날조였다.** 기억-인용과 도구-근거는 글자만 봐선 구별이 안 된다. 모델이 실제로 무엇을 조회했는지를 코드로 대조하는 장치만이 이 날조를 잡았다. 이 대조가 도구 측정 신뢰성의 전부다.

## 인접 벤치와 무엇이 다른가

한국·법률·세무 벤치를 실측 조사했다(KMMLU-Pro, KBL, LegalBench, PLAT). 이들은 대부분 객관식·분류 정답률을 잰다. K-TaxBench의 차별점은:

1. 객관식 정답률 → **생성형 실무 신뢰도** 5축 채점
2. **근거 강제 채점** — 조문·기준서 문단 단위로 인용 정확성을 코드 채점(인용 정확성을 직접 채점하는 벤치는 조사 범위 내 없었다)
3. **적시성** — 문항별 시점 버전 관리
4. (조사 범위 내) **유일한 생성형·다차원 한국 세무·회계 실무 벤치**
5. **오염 저항·self-eval 방지**

우연한 교차확인 하나: 한국 법률 벤치 KBL의 open-book 효과가 +8.6%로, 우리 RAG +8.6과 같은 크기였다. 독립된 벤치에서 retrieval 효과가 재현된 셈이다. (단, 이건 점수 직접 비교가 아니라 *설계·평가축 비교*다.)

## 지금 상태와 솔직한 한계

공개 리더보드가 라이브다 — **https://tax-benchmark.askewly.com**. 읽기 전용 뷰어로 랭킹·차원별 히트맵·대표 오류 사례를 보여주고, 리더보드 왜곡(재시도 best-pick·선택적 철회 등)을 막는 제출 정책을 UI로 강제한다.

솔직한 한계: 공개 34문항에 대한 **본인 검수 라운드를 마쳤고**(2026-06-12), 다음은 외부 세무·회계 전문가 검수다. 전문가 검수를 통과한 문항만 정식 버전으로 승격한다. LLM-judge에는 런마다 점수 분산이 있어 작은 실험들의 결론은 "방향성"으로 한정한다. 1차 범위는 Claude 계열 3모델에 집중했고, GPT·Gemini 교차 변별은 다음이다.
