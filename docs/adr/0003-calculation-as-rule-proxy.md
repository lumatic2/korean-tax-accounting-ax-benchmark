# 0003 — 계산문항은 "룰 프록시"로만 채택, 순수 산수는 의도적으로 최소화한다

## Status
Accepted (2026-06-02)

## Context
초기 문항셋은 `calculation` 유형을 6/20(25%)로 두었다. 그러나:
- 실제 평가 대상은 *도구를 가진 AI 시스템*(AX 제품 = agent + 계산 코드)이다. 형제 레포 `tax-agent`만 봐도 세무 계산은 이미 파이썬 코드로 결정론적으로 처리된다 → **산수는 풀린 문제**다.
- M3 1차 파일럿에서 모델 변별(spread 36)은 **환각·근거·판단**에서 났고, 산수에서 나지 않았다. 법인세 haiku 붕괴 원인은 *가짜 조문 6건*이었다.
- 설계 원리 §1.1(구성 타당도)·§5(Saturation·구성 무효 안티패턴)는 이미 "산수형은 변별 못 함"을 경고한다.
- 그럼에도 산수 채점은 *싸고 결정론적*(judge·self-eval 리스크 없음, §2.2)이라는 진짜 장점이 있다 — 버리기 아깝다.

선택지:
- **(A) 계산문항 비중 유지** — 결정론 채점 자산이지만 saturation, 구성 타당도 약화.
- **(B) 계산문항 전면 폐지** — 결정론 앵커를 잃음.
- **(C) "룰 프록시"로만 한정** — 숫자가 *룰 숙련도(한도·율·요건·조건분기)의 프록시*일 때만 채택, 순수 산수는 배제.

## Decision
**(C).** 계산문항 채택 기준을 단일 질문으로 고정한다: **"룰을 아는 전문가도 룰 선택을 잘못해서 틀릴 수 있는가?"** YES면 채택(숫자=결정론 앵커), 산수만 물으면 배제. 목표 비중 ~10%로 낮추고, 무게중심을 citation(근거)·case·risk·agent로 옮긴다. 상세 비중·기준은 [question-blueprint.md](../question-blueprint.md).

## Consequences
- ✅ 구성 타당도↑(점수가 실무 신뢰도를 대변). Saturation 회피.
- ✅ 결정론 채점 앵커는 유지(룰프록시 계산 = 코드 채점 가능 + 변별).
- ✅ M4 방법론 리포트에 "왜 산수를 적게 냈는가"의 근거 자산 확보.
- ⚠ 잘못 분류된 기존 계산문항 정리 필요 — `vat-0002`(→risk_analysis 재분류), `corp-tax-0002`(→룰프록시 계산으로 재작성)로 이번에 처리.
- ⚠ 계산문항 채택 판단이 주관적일 수 있음 — blueprint §2의 4겹 기준으로 형식화.
