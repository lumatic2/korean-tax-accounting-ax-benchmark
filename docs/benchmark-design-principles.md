# 벤치마크 설계 원리 (K-TaxBench)

> "훌륭한 벤치마크의 공통점"을 K-TaxBench에 박은 문서. 모든 문항 제작·채점·리더보드 설계 결정은 이 문서를 인용한다.
> 상태: **v0.2 (2026-06-01)** — 1차 리서치 반영. 외부 벤치마크 채점 방식·개념체계 문단 번호 검증 완료(§근거). 남은 미해결은 §6.

---

## 0. 핵심 테제

벤치마크 점수는 그 자체로 **하나의 정보**다. 회계법인이 "이 점수를 믿고 벤더를 고른다"가 성립하려면, 점수가 **타당·재현·게이밍 저항**을 만족해야 한다.

그런데 회계는 이미 "신뢰할 수 있는 정보"의 조건을 정의해 두었다 — 「재무보고를 위한 개념체계」의 **질적 특성**이다. K-TaxBench는 이 프레임을 *벤치마크 설계 원리로 차용*한다.

> **재무정보에 요구하는 질적 특성을, AI 평가 정보에도 동일하게 요구한다.**

이것이 이 벤치마크가 일반 ML 벤치마크와 다른 출발점이며, "회계 전문가가 설계한 AI 평가"라는 정체성의 뿌리다.

---

## 1. 근본적 질적 특성 → 벤치마크의 존재 조건

개념체계의 두 **근본적 질적 특성**(문단 2.5~)이 충족되지 않으면 벤치마크는 아예 무의미하다. 보강적 특성으로 메울 수 없다 — 개념체계도 "근본 특성이 먼저 있어야 보강 특성이 효과를 낸다"고 명시한다.

### 1.1 목적적합성 (Relevance, 문단 2.6) → 구성 타당도 (Construct Validity)

- **회계 정의**: 정보가 이용자의 의사결정에 차이를 만들 수 있어야 한다.
- **벤치마크 적용**: 점수 1점의 차이가 "실무에 1만큼 더 믿고 맡길 수 있음"을 의미해야 한다. 객관식 정답률이 높다고 세무조정을 맡길 수 있는 게 아니다 — 측정 대상(점수)이 측정하려는 능력(실무 수행)의 충실한 프록시여야 한다.
- **외부 근거**: LegalBench(162개 과제, 6종 법적추론)는 *법률 전문가가 과제 설계를 주도*해 "실제 법률 업무에서 가치 있는 능력"을 측정하도록 했다. 우리도 동일 — 문항이 실무를 대변해야 한다.
- **설계 결정**:
  - 문항 유형을 실무 작업의 프록시로 설계 (사례 판단형·근거 제시형·agent형 비중 확보).
  - 단순 지식 암기형은 *기초 변별*용으로만, 실무 타당도는 사례·agent형이 담당.
  - 각 문항에 "이 문항이 측정하는 실무 역량" 태깅 (positioning.md 5대 역량과 연결).
  - **유형별 구성 비중·계산문항 채택 기준은 [question-blueprint.md](question-blueprint.md)로 고정**(M1 확충 시험 명세서). 계산은 "룰 프록시"로만 채택 → [adr/0003](adr/0003-calculation-as-rule-proxy.md).

### 1.2 표현충실성 (Faithful Representation, 문단 2.12) → 전문가 검수 정답 (load-bearing ②)

- **회계 정의**: 정보가 나타내려는 경제적 실질을 완전·중립·무오류하게 표현해야 한다.
- **벤치마크 적용**: 정답(ground truth)이 **법령·기준서의 실질을 충실히 표현**해야 한다. 정답이 틀린 벤치마크는 측정이 아니라 오염원이다.
- **외부 근거**: GPQA는 **전문가 작성 → 전문가 검증 → 비전문가 테스트** 파이프라인으로 정답 신뢰도와 난이도를 동시에 증명했다(비전문가 정답률을 측정해 "검색해도 못 푼다"를 입증). SteuerEx(독일 세법)는 **115문항을 전문가 검증**으로 구성했다.
- **설계 결정**:
  - 모든 정답·해설·근거는 **세무사/회계사 검수**를 거친다 (검수 없는 문항은 비공개셋 편입 금지).
  - 정답에 **기준일 도장** 필수 (`기준일: YYYY-MM-DD 시행 법령 기준`) — 세법은 시점 종속적이라, 기준일 없는 정답은 표현충실성 위반.
  - GPQA식 **비전문가 베이스라인** 측정으로 문항 난이도·변별력을 입증(선택).
  - "완전·중립·무오류": 사실관계 누락 없이 제시, 특정 결론으로 유도하지 않음, 검수 라운드로 오류 제거.

## 2. 보강적 질적 특성 → 벤치마크의 신뢰도 4축

근본 특성이 충족된 위에서, 네 **보강적 질적 특성**(문단 2.23~)이 벤치마크를 *실제로 쓸 수 있게* 만든다. (사용자 정의: 이 넷이 가장 근본적으로 중요.)

### 2.1 비교가능성 (Comparability, 문단 2.24)

- **회계 정의**: 이용자가 항목 간 유사점과 차이점을 식별·이해할 수 있게 한다.
- **벤치마크 적용**: 모델 A와 모델 B를 **같은 조건에서** 재서 줄 세울 수 있어야 한다. 조건이 다르면 점수 비교가 거짓말이 된다.
- **외부 근거**: SWE-bench 점수는 **스캐폴드(평가 프레임)에 크게 의존**한다 — "Opus + Claude Code"와 "Opus + 커스텀 스캐폴드"가 다르므로 평가 프레임을 항상 명시해야 한다. 즉 *조건을 고정·기록하지 않으면 비교 불가*.
- **메커니즘**:
  - 고정 프롬프트 (모델별 프롬프트 튜닝 금지, 또는 튜닝 허용 시 명시).
  - **버전 핀**: 모델 버전·날짜, 데이터셋 버전, 평가 하네스·스캐폴드 버전을 점수에 함께 기록.
  - 모드 분리 비교 (closed-book / RAG / agent 를 섞어 비교하지 않음 — 같은 모드끼리만).
  - 리더보드 표준 컬럼 (모드·기준일·데이터 버전·스캐폴드·총점·분야별 점수).

### 2.2 검증가능성 (Verifiability, 문단 2.30) — load-bearing ① (오염저항) 포함

- **회계 정의**: 서로 다른 관찰자가 독립적으로 "정보가 실질을 충실히 표현한다"에 합의할 수 있어야 한다.
- **벤치마크 적용**: 누가 채점해도 같은 점수가 나와야 하고, 점수가 **학습 누출이 아니라 실제 능력**에서 나왔음을 보장해야 한다.
- **채점 정책 (확정)**:
  - 자동채점 가능한 건 코드로 (계산·객관식). SWE-bench가 *테스트 통과 여부*로 객관 채점하듯, 수치·정답은 프로그램 채점.
  - 근거·사례형은 **LLM-judge + 전문가 스팟체크**. 평가당하는 모델이 자기 답을 채점하지 않는다 (self-eval 천장 금지). LLM-judge 의 bias(self-preference·verbosity·position) 통제 + 표본 전문가 재검.
  - **부분점수 채점**: SteuerEx의 **statement-level partial-credit**(문장 단위 부분점수, 실제 시험 채점을 모사)를 차용 — "결론은 맞고 근거 일부만 맞음"을 점수로 표현.
  - **사례형 분해**: LegalBench의 **IRAC**(Issue 쟁점 / Rule 법규 / Application 적용 / Conclusion 결론)로 사례 답안을 분해 채점 → 우리 6차원과 정합.
  - **agent 모드 신뢰도**: τ-bench의 **pass^k**(k회 반복 중 성공 확률) 차용 — 1회 정답이 아니라 *반복 일관성*을 측정. (τ-bench 기준 SOTA도 pass^8 < 25%로 매우 불안정 → agent 신뢰도는 반드시 다회 측정.)
- **오염 저항** (load-bearing ①, 가장 치명적):
  - 공개 샘플 ↔ 비공개 검증셋 분리 (§4).
  - **post-cutoff 신규 문항**: 모델 학습 시점 이후 개정세법 기반 문항 = 오염 불가 + 최신성 동시 충족. (SWE-bench도 오염 우려로 Verified/Pro 후속이 나옴.)
  - canary 문자열, 비공개셋 주기적 로테이션, 제출 횟수 제한.
  - 근거 인용을 채점 차원으로: 답뿐 아니라 조문·판례·기준서 인용의 정확성을 따로 검증.

### 2.3 적시성 (Timeliness, 문단 2.33)

- **회계 정의**: 정보가 의사결정에 영향을 줄 수 있는 시점에 이용 가능해야 한다.
- **벤치마크 적용**: 세법은 매년 바뀐다. **현행 법령 기준으로 유효한** 문항·정답이어야 하고, 개정 시 빠르게 반영해야 한다. 작년 기준 정답은 올해 오답일 수 있다.
- **메커니즘**:
  - 문항·정답마다 기준일, 문서 자체에 개정 이력.
  - **버전 관리**: v2025 / v2026 등 연도(또는 개정)별 스냅샷. 법 개정 → 영향 문항 재검수 → 새 버전.
  - 신선도 파이프라인: 개정세법 모니터링 → post-cutoff 문항 정기 추가 (적시성 + 오염저항 + 변별력 동시 확보).
  - "정답이 시점에 따라 달라지는 문제" 관리는 리스크가 아니라 *해자* — 신선한 문항을 계속 못 만드는 경쟁자는 따라올 수 없다.

### 2.4 이해가능성 (Understandability, 문단 2.34) — load-bearing ③ (다차원 채점) 포함

- **회계 정의**: 합리적 지식을 가진 이용자가 이해할 수 있도록 표시·분류해야 한다.
- **벤치마크 적용**: 점수의 최종 소비자는 **회계법인 의사결정자**다. "총점 78점"만으로는 못 산다. *어디가 약한지*를 이해할 수 있어야 도입·개선 결정을 한다.
- **외부 근거**: HELM은 단일 지표 대신 **7개 지표 다차원 설계**로 평가 커버리지를 17.9%→96%로 끌어올렸다. 단일 점수는 이해가능성이 낮다.
- **메커니즘** (load-bearing ③ 다차원 채점):
  - 단일 정답률이 아니라 **6차원 분해 채점**: 결론 정확성 / 근거 정확성 / 계산 과정 / 사실관계 반영 / 불확실성 처리 / 실무 활용성 (positioning.md·rubric 과 연결).
  - 진단 리포트: 분야별(부가세·법인세·소득세·회계처리) × 차원별 점수 + 대표 오류 사례(환각·계산오류·근거오류).
  - "답은 맞지만 근거가 틀린 경우"를 분리해 보여주는 것 — 이게 일반 모델 벤치(정답률 하나)를 넘는 K-TaxBench 의 판매 포인트.

## 3. Load-bearing 3축

위 질적 특성 안에 박혀 있지만, 의사결정 시 항상 먼저 따질 세 축:

1. **오염 저항** (검증가능성 안) — 점수 신뢰의 생사. 공개/비공개 분리 + post-cutoff 신규문항.
2. **다차원 채점** (이해가능성 안) — 차별점이자 매출 근거. 어디가 약한지 진단.
3. **전문가 검수 정답** (표현충실성) — 도메인 전문성의 자산화. 기준일 도장.

> 새 문항·새 채점 방식·새 기능을 설계할 때: "이게 위 3축을 강화하는가, 훼손하는가?"를 먼저 묻는다.

## 4. 공개/비공개 하이브리드 구조

| | 공개 샘플 | 비공개 검증셋 |
|---|---|---|
| 비중 | 10~20% | 80~90% |
| 목적 | 방법론 시연·리더보드 인지도·마케팅 | 진짜 점수·유료 리포트·인증 배지 근거 |
| 오염 | 시간 지나면 오염 *전제* → "연습문제" 취급, 주기 교체 | 오염 통제·전문가 유지보수·로테이션 |
| 접근 | 공개 | 제출 횟수 제한·접근 통제 |

**리더보드 운영 경고 (실제 사례 학습)**: 「The Leaderboard Illusion」(2025)은 LMArena에서 *일부 공급자가 비공개로 여러 변형을 테스트하고 최고점만 공개*하는 selective disclosure 가 순위를 왜곡함을 보였다. → 우리 리더보드는 **제출 정책 공개, 변형 fishing 차단(제출 횟수·재시도 제한), 점수 철회 규칙 명문화**가 필요하다. 비공개 평가 서비스(Scale SEAL 등)는 이를 사설 서비스로 우회하는 모델.

## 5. 안티패턴 (피해야 할 실패 모드)

| 실패 | 증상 | 위반 질적특성 | 방어 |
|---|---|---|---|
| Saturation | 다 만점 → 변별 0 | 목적적합성 | 난이도 구배 + 헤드룸 |
| Contamination | 학습 누출로 점수 뻥튀기 | 검증가능성 | 비공개셋 + post-cutoff |
| Goodhart/gaming | 시험에만 최적화 | 목적적합성 | 제출제한 + 로테이션 |
| Selective disclosure | 변형 fishing 후 최고점만 공개 | 비교가능성·검증가능성 | 제출정책 공개·철회규칙 |
| 구성 무효 | MCQ 가 실무 못 대변 | 목적적합성 | agent·사례형 비중 |
| 채점 불안정 | judge bias·루브릭 모호 | 검증가능성 | 자동채점 우선 + bias 통제 + 전문가 스팟체크 |
| Agent 비일관성 | 1회 운 좋게 통과 | 검증가능성 | pass^k 다회 측정 |
| 정답 오류 | 법 개정으로 정답이 틀림 | 표현충실성·적시성 | 기준일 도장 + 검수 + 버전 |
| 통계 빈약 | 문항 적어 점수 노이즈 | 검증가능성 | 분야별 충분 문항수 + 신뢰구간 (§6) |
| 진단 부재 | 총점만, 어디 약한지 모름 | 이해가능성 | 다차원 분해 리포트 |

## 6. 외부 벤치마크 레퍼런스 (1차 리서치, 2026-06-01)

| 벤치마크 | 도메인 | 차용할 점 | 출처 |
|---|---|---|---|
| **GPQA** | 과학 QA | 전문가작성→전문가검증→비전문가테스트 파이프라인; "검색해도 못 풂" 난이도 증명; Diamond 198문항 | arXiv/2층 |
| **SWE-bench** | 코드 수정 | 테스트 통과로 객관 채점; **스캐폴드 의존 명시 필수**; 오염→Verified/Pro 후속 | swe-bench |
| **HELM** | 다목적 | **7지표 다차원** 평가(커버리지 17.9%→96%) | Stanford CRFM |
| **LegalBench** | 법적추론 | **IRAC 분해 채점**; 162과제·법률전문가 주도; 실무역량 측정 | arXiv 2308.11462 |
| **τ-bench** | 도구-에이전트 | **pass^k 신뢰도**; DB 종료상태 vs 목표상태 비교; SOTA도 불안정 | arXiv 2406.12045 |
| **SteuerEx / SteuerLLM** | 독일 세법 | **statement-level 부분점수**; 115문항 전문가검증; 6개 세법영역 | arXiv 2602.11081 |
| **NitiBench-Tax** | 태국 세법 | 실무 세법 사례; RAG vs long-context 비교; 조문 chunking·cross-reference | arXiv 2502.10868 |
| **DeonticBench** | 규칙기반 추론(미연방세 등) | 세법을 **의무/허용/금지(deontic)**로 모델링; **Prolog 실행 trace로 검증**(reference 프로그램 공개); 6,232과제, hard-subset SOTA 44~47% | arXiv 2604.04443 |
| **CFinBench** | 중국 금융·회계 | **4영역 커버리지 청사진**(지식/자격/실무/법규)·43세부; 99,100문항이나 객관식 3유형(MCQ 편중=구성타당도 약점) | arXiv 2407.02301 |

> **K-TaxBench 위치**: SteuerEx·NitiBench-Tax·DeonticBench 가 가장 가까운 형제(국가별 세법 벤치). 우리 차별점 = ① 시험문제를 넘어 **판례·실무사례·agent 과제**까지 ② **회계 개념체계 기반 다차원 채점** ③ 공개/비공개 하이브리드 + 인증 배지(제품화).
>
> **두 흡수 포인트**: (a) DeonticBench의 *deontic 프레임* — 세무 결론은 본질적으로 "해야 한다/할 수 있다/하면 안 된다"라, 문항·루브릭을 이 축으로 분류하면 근거 검증이 형식화된다. *Prolog trace*까지 갈진 미정이나, 최소한 **계산·세무조정의 단계 trace를 검증 대상으로** 삼는 아이디어는 검증가능성을 강화. (b) CFinBench의 *4영역 커버리지 청사진*은 우리 분야 blueprint(시험 명세서)의 참고틀 — 단, CFinBench의 MCQ 편중은 우리가 *피하는* 방향(구성타당도).

## 7. 미해결 / 다음 리서치 (judge 규약: 검증 전 사실 인용 금지)

- **최소 문항 수**: 고품질 벤치는 과제당 수십~수백 문항(GPQA Diamond 198). 한 position paper는 **데이터 수백 개 미만에서 CLT(정규근사) 사용 경고** → MVP 50~100은 *파일럿*으로 OK이나, **분야별 신뢰구간을 주장하려면 주요 분야(부가세·법인세·소득세)별 ~100+ 필요**. 정확한 표본설계는 채점 분산 측정 후 확정. **[부분 확인]**
- 한국 **기출문제·해설 저작권** 및 법령·판례 공공데이터 라이선스 → `data-strategy.md` 에서 별도 조사.
- 한국 Big4 AI 조직이 실제로 살 **검증 리포트 형태** → 고객 디스커버리.
